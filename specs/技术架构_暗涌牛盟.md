# 技术架构与核心算法设计：【暗涌牛盟】

## 1. 技术栈选型
基于已确立的全栈化改造路线：
- **前端**：Vue 2 / Vue 3 + Vuex + vue-router
- **视觉/动效**：CSS3 (Neon effects), HTML5 Canvas (搓牌动画), `navigator.vibrate` (H5震动API), `html2canvas` (长图生成)
- **网络通信**：原生 WebSocket + msgpack-lite (保持二进制高频通信，低延迟)
- **后端**：C++ (核心发牌/控牌引擎) 或 Node.js (房卡/后台逻辑)。*注：目前后端主逻辑在 C++ 游戏服，以下设计主要描述游戏服的改造。*
- **数据库**：MySQL (持久化对局记录、抽水账单), Redis (预发牌池缓存、实时权重配置)

## 2. 核心系统设计

### 2.1 房卡与 0点起步账本 (Room Card & Ledger)
- **房卡生成**：后端生成 6 位随机 alphanumeric Hash (如 `7A92K1`)，存入 Redis，设置 TTL 或局数计数器 (0/20)。
- **无底洞账本**：玩家入局时 `gold = 0`。每局结算时，计算理论输赢。
- **1%抽水公式**：
  ```javascript
  if (winGold > 0) {
    let actualWin = Math.round(winGold * 0.99 * 100) / 100; // 保留两位小数
    let tax = winGold - actualWin;
    // tax 归入房主抽水池
  }
  ```

### 2.2 交互层设计 (UI/UX)
- **视觉风格**：`#121212` (深金属拉丝底), `#E6A23C` (金), `#F56C6C` (霓虹红)。
- **布局划分**：
  - **2人局 (Duel)**：上下分屏。中间展示局数 `Round: X/20` 和 `Tax: YYY`。血条组件（基于正负分数渲染红绿长度）。
  - **5人局 (Club)**：环形排布。庄家头像添加 CSS `box-shadow` 和 `animation` 模拟雷电特效。
- **搓牌动画 (Squeeze Card)**：
  - 使用 Canvas 或 CSS `clip-path` + `transform: rotateX/Y`。
  - 监听 `touchstart`, `touchmove` 计算滑动距离，动态改变扑克牌背面的遮罩区域，露出底部的数字 1/4 到 1/2。
  - 当露出关键点时调用 `if (navigator.vibrate) navigator.vibrate([50, 50, 50]);` 触发短促高频震动。

### 2.3 暗箱控牌引擎 (Manipulation Core) ⭐️ 核心难点
这是本项目的灵魂，必须保证毫秒级拦截且绝对不穿帮（无重复牌）。

#### 2.3.1 预发牌池 (Pre-deal Pool)
每局开始 (State: WaitStart -> Deal4)，后端生成 52 张牌的洗牌数组。
分配：4张明牌 * N人 + 预留第5张暗牌 * N人。剩下的存入 `RemainingDeck`。

#### 2.3.2 拦截器与实时换牌 (Interceptor & Hot-swap)
**触发时机**：发送第 5 张牌给客户端前 (State: Betting -> Deal1)。
**流程**：
1. **Query Weight**：查 Redis 获取房主的干预指令（例如：玩家A [强制牛牛]，玩家B [强制无牛]）。
2. **Calculate Target**：
   - 玩家A 前 4 张是 [♦2, ♣3, ♥5, ♠K] (总和 10, K算10)。
   - 要凑牛牛，第 5 张必须是 10 的倍数（10, J, Q, K）。
3. **Hot-swap 算法**：
   ```python
   Card find_card_for_target(Player p, TargetType target, Deck remaining) {
       // 遍历 remaining 牌堆
       for (Card c : remaining) {
           if (evaluate_niu(p.hand4, c) == target) {
               remaining.remove(c);
               remaining.add(p.pre_deal_5th_card); // 把原本预发的废牌扔回牌堆
               return c;
           }
       }
       // 如果找不到完美的（极低概率），降级寻找次优解（如牛9）
   }
   ```
4. **平滑校验**：由于是从 `RemainingDeck`（去除了场上所有明牌）中挑选，绝对不可能出现两张一模一样的牌（如两张♠K），保证了“表面公平性”。

#### 2.3.3 房主上帝面板 (Master Control)
- 独立于游戏客户端的一个 WebSocket 频道或 Vue 悬浮窗（仅对 Owner 鉴权可见）。
- **实时透视**：后端在 `Deal4` 阶段就将预发的第 5 张牌通过私有协议发给房主客户端。
- **干预接口**：房主点击 [全场点杀 玩家B]，前端发送 `MsgMasterControl { target: B, action: 'KILL' }`。后端收到后，将 B 的权重设为 `ForceLose`，并在拦截器中为 B 匹配最小的牌型，为庄家匹配最大的牌型。

### 2.4 总结算与防伪系统 (Final Report)
- **长图生成**：20 局结束时，前端路由跳转至 `/report` 页面，渲染 20 局的总榜单（净分、抽水、连胜）。
- **二维码**：调用 `qrcode` 库，将包含对局 Hash 的 URL (`https://xxx.com/replay/7A92K1`) 生成二维码贴在页面底部。
- **截屏导出**：使用 `html2canvas` 将整个 DOM 转为 Base64 图片供用户长按保存。
- **微信通知**：后端在销毁房间前，调用 Server酱 / 微信公众号 API，发送模板消息：“房间 7A92K1 已结算，总抽水积分：XXX”。
